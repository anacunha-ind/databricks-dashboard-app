Bloco 1 · Marketplace

Estrutura e Componentes

O repositório é o marketplace. Nenhum clone necessário — o Claude Code busca os plugins pela URL do remoto.

claude_code/
├── .claude-plugin/marketplace.json
├── plugins/
│   ├── commons/
│   ├── analytics-engineering/
│   ├── data-engineering/
│   └── migration-engineering/
└── QUICKSTART.md
Cada plugin de função serve um perfil: Analytics (dbt/SQL), Data (Databricks/PySpark) ou Migration (notebook → dbt). Instale apenas o seu.
Skills
Ativadas automaticamente. Peça "adicione testes" → skill dbt-tests aplica as regras do projeto.
Agentes
Subprocessos especializados (analytics-engineer, dbt-auditor, notebook-migration) — rodam isolados.
Comandos
Slash commands com fluxo predefinido: /review, /review-pr, /document-pr.
MCP Servers
Confluence MCP ativa automaticamente — leitura de páginas direto na sessão sem copiar e colar.
Bloco 1 · Marketplace

Instalação em 4 Passos

Guia completo em QUICKSTART.md.

1
Configurar credenciais em ~/.env
export BITBUCKET_TOKEN="..." && export JIRA_API_TOKEN="..." && source ~/.env
2
Registrar o marketplace
/plugin marketplace add git@bitbucket.org:indiciumtech/claude_code.git
3
Instalar o plugin do seu perfil
/plugin install analytics-engineering
Troque pelo plugin correto: data-engineering ou migration-engineering.
4
Reiniciar e verificar
/exit → claude → /plugin → /mcp
Plugin na aba Installed + servidor MCP confluence conectado.
Credenciais necessárias
BITBUCKET_TOKEN — HTTP Access Token (não App Password)
Gerar em account/settings/access-tokens/
JIRA_API_TOKEN — token sem escopo selecionado
Gerar em id.atlassian.com/manage-profile/security/api-tokens
CONFLUENCE_URL — URL base do Confluence
Ver .example.env no repositório para o bloco completo
Para o Confluence MCP: instale o uv e defina as três variáveis (CONFLUENCE_URL, JIRA_EMAIL, JIRA_API_TOKEN) antes de abrir o Claude Code. O servidor inicia automaticamente.

Bloco 2 · Skill Creator

Instalando o Skill Creator

Skill oficial da Anthropic que guia a criação de forma interativa, com testes automáticos de qualidade.

1
Adicionar o marketplace da Anthropic
/plugin marketplace add https://github.com/anthropics/skills
2
Instalar o plugin document-skills
/plugin add document-skills
Fica disponível em todas as sessões futuras.
3
Abrir o Claude Code no repositório do marketplace
cd claude_code && claude
4
Ativar o Skill Creator
/document-skills:skill-creator
O Claude guia por: nome, domínio, description e conteúdo.
O Skill Creator vai perguntar
Nome da skill — em kebab-case ex: dbt-freshness, pyspark-profiler, ingestion-validator
Domínio — subpasta em skills/ ex: dbt, databricks, bitbucket, sql
Description — o gatilho de ativação automática Frase em português: quando e por que o Claude deve ativar
Conteúdo — instruções, padrões, exemplos O que o Claude deve fazer quando a skill for ativada
Após criar, o Skill Creator gera casos de teste automáticos e avalia se o trigger está bem calibrado antes de salvar o arquivo.
Bloco 2 · SKILL.md

Anatomia + Boas Práticas

O frontmatter controla o gatilho. Uma description imprecisa faz a skill nunca ativar — ou ativar na hora errada.

---
name: dbt-freshness
description: Ativa quando o usuário
quer configurar freshness em
sources dbt ou monitorar recência
de dados upstream. Não ativa
para testes genéricos de dado.
---

# dbt Freshness

## Visão Geral
Configure freshness em sources dbt para monitorar recência dos dados.

## Fluxo de Execução
1. Ler dbt_project.yml
2. Localizar arquivo de sources
3. Adicionar bloco freshness

## Exemplos
"configure freshness de 6h" → warn_after: {count: 6, period: hour}
✓ Faça
Triggers explícitos: "Ativa quando X, Y ou Z"
Description em português
Anti-triggers: "Não ativa quando…"
Referencie sstyle-guides/ em vez de copiar style guides inteiros
Lazy loading: leia templates só na fase que os usa
Abaixo de 500 linhas
✗ Evite
Description genérica: "Ajuda com dbt"
Copiar style guides completos no corpo
Sobreposição com skills existentes
PII ou credenciais no conteúdo
Regras de negócio proprietárias literais
Esquecer de registrar no plugin.json
Registrar no plugin.json:
"skills": ["./skills/<dominio>"]
O Claude Code descobre todos os SKILL.md dentro do diretório automaticamente.

Bloco 3 · Pull Request Reviewer

Pipeline e Escala de Severidade

O revisor executa uma vez por PR. Sev 3–5 viram comentários inline; Sev 1–2 aparecem só no sumário.

Fluxo do Pipeline
①
PR aberto → pipeline acionado no Bitbucket
②
Bot já comentou? → encerra sem ação
③
PR trivial (< 20 linhas) → ignora
④
Lê CLAUDE.md + review-context.md → calibra rubrica
⑤
Posta comentários inline (Sev 3–5) + sumário
⑥
Aprova automaticamente se [APPROVED]
Indicadores de Risco
[HIGH RISK] Achados Sev 5 presentes
[REQUIRES ATTENTION] Achados Sev 4
[APPROVED WITH CAVEATS] Apenas Sev ≤ 3
[APPROVED] Sev 1–2 ou sem achados
Escala de Severidade
Sev 5
Crítico — Bloqueador
Credencial exposta, DELETE sem WHERE, bypass de auth, violação LGPD/SOX. Correção obrigatória antes do merge.
Sev 4
Alto
Bug em funcionalidade central, N+1 queries, ponteiro nulo sem tratamento, PII em log. Correção fortemente recomendada.
Sev 3
Médio
Dívida técnica, função longa, nova funcionalidade sem testes. Recomendado — pode ser PR subsequente.
Sev 2
Baixo
Legibilidade, nomes de variáveis, refatoração leve. Melhoria sugerida — backlog.
Sev 1
Observação
Nitpick estilístico. Opcional — a critério do autor.
Bloco 3 · Customização

Customizando a Rubric

Dois arquivos opcionais fazem o revisor conhecer as convenções do seu repositório.

1. CLAUDE.md — Convenções gerais
Se o projeto já usa Claude Code, provavelmente já existe um CLAUDE.md. O revisor lê automaticamente — nenhuma configuração extra. Convenções declaradas nele elevam a severidade de achados quando violadas.
2. .claude/review-context.md — Overrides
Exclusivo para o revisor automático. Complementa o CLAUDE.md sem poluí-lo com instruções de CI.
Template: plugins/commons/templates/review-context.md
O que configurar em review-context.md
Stack e linguagens — prioriza categorias relevantes para o PR
Convenções obrigatórias — violações elevam a severidade
Boas práticas a reconhecer — menção positiva no sumário
Arquivos e padrões a ignorar — além dos lockfiles da rubrica genérica
Ajustes de severidade por área — escalar em módulos críticos; reduzir em experimentais
Ambos os arquivos são opcionais. Repositórios sem eles recebem revisão com a rubrica genérica de engenharia de software.
